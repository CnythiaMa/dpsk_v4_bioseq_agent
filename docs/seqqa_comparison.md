# SeqQA2 对比 · 四方(flash / pro × 无工具 / 代码 Agent)

> 数据集:LabBench2 `seqqa2`(400 题,20 类 × 20)· 官方程序化验证器评分
> 日期:2026-06-06 · **本文为 SeqQA 的完整版对比**(覆盖 4 次实验)

| 配置                             | 模型怎么拿到序列                     | 怎么算               | 怎么出答案                                         |
| -------------------------------- | ------------------------------------ | -------------------- | -------------------------------------------------- |
| **flash 无工具**           | **inject**:序列全文贴进 prompt | 模型 in-context 心算 | 直接 `<answer>`                                  |
| **pro 无工具**             | **未注入**(只给题面)⚠️       | —                   | 直接 `<answer>`                                  |
| **flash / pro 代码 Agent** | 用 `run_python` 自己读文件         | Biopython 精确计算   | 模型自己 `<answer>`(**无 compose 格式层**) |

- **flash 无工具**用的是官方 inject harness(`results/seqqa_base/results_full.json`),口径干净。

---

## 总览

| 配置                        |          当前正确数 | CoT 截断强制返回 | Agent 截断(待补刷) | 未作答·无数据(待补刷) |    已作答准确率 |
| --------------------------- | ------------------: | ---------------: | -----------------: | ---------------------: | --------------: |
| flash · 无工具(inject)     |           171 / 400 |               18 |                 — |                     — |           44.8% |
| flash · 代码 Agent         |           155 / 400 |               — |                168 |                     — | **66.8%** |
| pro · 无工具 ⚠️(残缺)    |            63 / 324 |               — |                 — |                    211 |           55.8% |
| **pro · 代码 Agent** | **203 / 400** |               — |                146 |                     — | **79.9%** |

> **三类 `no_answer` 的成因(每个配置只会落在其中一列)**:
>
> - **CoT 截断强制返回**:无工具时,模型在**长序列**上陷入思维链反复数碱基,输出被截断/没收敛,最终没出现 `<answer>` 标签。这是模型本身没算完,**非简单重跑可补**。
> - **Agent 截断(待补刷)**:代码 Agent 在 **6 轮上限**内没收敛,强制收尾那轮没按 `<answer>` 格式吐。**提高轮次上限 / 补一层格式对齐即可补刷回来。**
> - **未作答·无数据(待补刷)**:该残缺跑**根本没给模型序列**(既没注入也没工具),无数据可算 → 非格式输出。**用正确设置(注入或工具)重跑即可补刷。**
>
> **已作答准确率** = 当前正确数 ÷ (总数 − 上面三列之和),剔除"没给出可提取答案"的题,最能反映**模型实际算得准不准**。

## 真正的结论(和直觉相反,务必看清)

### 1. 无工具(inject)本身就不弱

flash 无工具把序列**全文贴进 prompt**就能拿 **42.8%**。原因:SeqQA 很多题的序列**很短**(如 `ATCGATCG`、`MKTLLLTLVV`),模型 in-context 直接就能数对。所以"工具 vs 无工具"在 SeqQA 上**不是 0 vs 1 的关系**。

### 2. 工具真正的增益:把"算得准"几乎翻倍——但被格式问题掩盖

看**已作答准确率**这一列,才是干净的"计算能力"对比:

|              | flash 无工具 | flash 代码Agent |   pro 代码Agent |
| ------------ | -----------: | --------------: | --------------: |
| 已作答准确率 |        44.8% | **66.8%** | **79.9%** |

**模型一旦用工具算、并且给出了答案,正确率从 ~45% 跳到 67%(flash)/ 80%(pro)。** 工具把"算得准"实打实地提上去了。

**但原始准确率没体现出来**:flash 代码 Agent 原始只有 38.8%(甚至低于无工具 42.8%),因为它**丢了 168 题在 `no_answer_extracted`**——模型算对了却**没按 `<answer>` 格式输出**(本轮**有意没接 compose 格式对齐层**)。

> **换算:若补上格式层、把已作答准确率铺到全 400 题**,flash ≈ 0.668×400 ≈ **267/400(67%)**,pro ≈ 0.799×400 ≈ **320/400(80%)**。
> **这 ~150 题的失分不是"算不对",是"格式没对齐",是工程一层就能补的。**

### 3. 工具的增益集中在"长序列"题型

分题型看(取干净的三列;`pro 无工具`残缺不列):

| 题型                                                       | flash 无工具 | flash 代码Agent | **pro 代码Agent** | 备注                                    |
| ---------------------------------------------------------- | :----------: | :-------------: | :---------------------: | --------------------------------------- |
| **gc_content**                                       |     4/20     |      17/20      |     **20/20**     | 长序列(rpsR 1315bp):心算挂,工具满分 ★  |
| **restriction_counts**                               |     4/20     |      17/20      |     **20/20**     | 长序列(rpoC):同上 ★                    |
| **primer_interactions**                              |     5/20     |      0/20      |     **19/20**     | pro 代码大胜(flash 没收敛)              |
| molecular_weight                                           |    11/20    |      6/20      |          12/20          |                                         |
| protein_hydrophobicity                                     |    14/20    |      15/20      |          18/20          |                                         |
| restriction_digest                                         |    19/20    |      13/20      |          19/20          | 无工具已会(序列短)                      |
| pairwise_distances                                         |    18/20    |      18/20      |          18/20          | 短序列,都会                             |
| enzyme_kinetics                                            |    18/20    |      17/20      |          19/20          |                                         |
| msa_scoring                                                |    14/20    |      10/20      |          14/20          |                                         |

## 一句话总结

> **SeqQA 上工具的价值是"把长序列的精确计算补满"(gc/restriction_counts 4→20),并把"算得准"的比率从 ~45% 提到 67–80%;但因为本轮去掉了 compose 格式层,~150 题算对了却没按 `<answer>` 输出,原始分被拉低,flash 甚至低于 inject 基线。补一层格式对齐是性价比最高的下一步(预计 flash→67%、pro→80%)。**

> 注:SeqQA 以本文为唯一口径。早前一版曾用**残缺的 pro 无工具跑(19.4%)**作基线、得出"近翻倍",并不准确;干净的无工具基线是 **flash inject 42.8%**(见上)。
