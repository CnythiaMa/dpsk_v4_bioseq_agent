01 — Background

我们想评估 DeepSeek-V4 在生物领域的表现。参照 OpenAI 的 gpt-rosalind，选用它评测时用到的两个开源榜单：

为什么选 LabBench

在两者之间，我们选了更偏序列任务的 LabBench：

  • BixBench 已经把 Agent 作为评测的内置元素（给模型工具/notebook 环境），优化空间相对有限；
  • LabBench（SeqQA / CloningQA）是「裸模型」序列任务，正是能清楚暴露模型短板、且有明确优化抓手的地方。

关键观察：失败模式

在涉及较长序列的任务上，比如需要核苷酸互补配对或核苷酸计数，模型很容易陷入思维链上冗长、反复的计数，而且数着数着就错（我们在多个案例里看到它编造序列、数错碱基、反向互补出错）。

而这些对数值敏感、靠规则可精确求解的任务，恰恰是工具一击即中的。

于是思路自然导向：用 Agent / 工具辅助来补这个短板。

▶CoT 冗余计数思考示例（点击展开）
String: g a c g a g c t g t a c a a g t a a a a g g g t g g g c g c g c c g a c c c a g c t t t c t t g t a c a a a g t g g t t

模型逐位编号（1 g, 2 a, 3 c, 4 g, 5 a, 6 g … 60 t），再手工推算坐标：

「Base 3589 is the 49th character (since 3589-3541+1 = 49). Index 49 = t. So base 3589 = t … The downstream sequence from 3589 for 25 bp would be bases 3589 to 3613 … bases 3589-3600: t a c a a a g t g g t t → "tacaaagtggtt" … 3601-3613: g a t c t a g t t c t a g … so the downstream sequence (25 bp) is "tacaaagtggttgatctagttctag".」

随后又进入 Sorcs2 插入片段的长串手工坐标推算（CDS join、ATG 起点、扩增引物 5′ 延伸），整段思维链被大量逐碱基计数占据，这正是工具一行代码即可替代的部分。

03 — Dead End

思路一（走过的弯路）：把 Agent 能力外置进 Prompt

LabBench 的题目限制使用 in-context learning 类方法，直接添加 Agent 能力未必符合题意。所以我们先尝试把 Agent 能力外置：用脚本预先算好核酸序列的结构化摘要信息，塞进 prompt 里，让模型「读现成的摘要」而不是自己数。

示例 A · SeqQA 注入长度、GC 含量、翻译、坐标等信息

%% [asserted] id=demo  organism="synthetic"  gene=demoG  molecule=DNA  topology=linear
%% [computed] length=40bp  GC=0.45  MW=12423.9  codon_table=1

Seq demo [1..40] 5'->3'
└─ Region CDS [9..32] demoG -> 8 aa   %% GC=0.4167 Tm=55.5C
    c1 ATG (M) [9..11] START
    c2 GCT (A) [12..14]
    c3 GAA (E) [15..17]
    c4 AAG (K) [18..20]
    c5 CTG (L) [21..23]
    c6 ATC (I) [24..26]
    c7 GCA (A) [27..29]
    c8 TAA (*) [30..32] STOP
%% complement strand implicit (A:T, G:C); reverse_complement on request

每个密码子的碱基、氨基酸、坐标、起止都外显；GC/Tm 也算好，模型不必再在思维链里数到第几位、翻译成什么。

示例 B · CloningQA 注入基因的结构化片段信息

*** 你唯一的工作数据是下面的 STRUCTURED SUMMARIES。不要去原始 GenBank 里手数坐标——
    在 ORIGIN dump 里手数位置是错引物的最大来源。每个特征的两端 45bp 边序列已预先算好。***

克隆只关心「特征的边」（CDS 起止、载体切点、重组位点），所以把每个特征两端各 45bp 的确切边序列 + 坐标 + GC 直接铺在前面，引物设计要的片段直接抄，不用再去 8911bp 的 ORIGIN 里数。

但最终发现这条路不行

  • SeqQA：把这种摘要塞进 prompt，等于直接泄露了答案（摘要里就含 GC/Tm/翻译等要算的量），失去评测意义；
  • CloningQA：边序列都铺好了，但复杂的多步克隆设计还是要模型自己做装配设计（选接头、保读框、多片段定序），光给静态摘要效果并不好。
证伪后，转向思路二：真正给模型工具调用能力。
04 — The Solution

思路二（最终方案）：给模型工具调用 / 代码执行能力

于是切换思路，给模型真正的工具调用能力，走 Agent 路线：不预先算好喂给它，而是给它一个沙箱，让它自己写代码、自己算、自己验证。核心文件：

工具环境 = 一个沙箱 + 一个自检

工具作用
run_python(code) 在题目目录起一个带状态的 Python 子进程（Biopython 可用，状态跨调用持久）。模型用它读文件、定位 feature、抠序列、做突变、算 GC/Tm/计数、设计引物/overhang。
dry_run_protocol(protocol) 把模型设计的克隆协议在硅基里模拟装配（PCR 退火、Gibson/GoldenGate/酶切），返回「能不能装配 / 错在哪」，交卷前的自检，不泄露标准答案。
沙箱本体不含任何生物学逻辑，它只是个执行器，真正的领域代码是模型每一轮现场写出来、当场执行、用完即弃的。智能在模型，算法在 Biopython，沙箱只提供「把模型写的代码跑起来」的轨道。
05 — Architecture

架构演进：从三层路由到「最小内核」

最初搭了一个三层 routed_pipeline（分类路由 → solver/agent → 格式对齐），先进行任务类型分类，后来发现只需要最核心的沙箱环境，所以去掉了冗余的其他层。

层原设计本轮是否使用
L1 路由分类器判断「纯计算走 solver / 还是要 LLM」去掉 · 每题直接进 Agent
L2 solver 分流命中确定性函数(gc/酶切计数)就直接算去掉 · 纯靠模型写代码
L2 代码 Agentrun_python + dry_run 多轮循环核心保留
L3 compose 格式对齐把答案套进 <answer> 格式去掉

最终最小内核 · labbench-code-agent/

sandbox.py   ← run_python 沙箱 + dry_run 自检(工具本体)
agent.py     ← 多轮 tool-calling 循环(design_cloning / solve_seqqa),记录完整 transcript
run_cloning.py / run_seqqa.py  ← 驱动 + 官方打分 + 落盘
llm.py / adaptive.py / config.py  ← 请求层(429 重试)/ 自适应并发 / 配置

08 — Dataset Issues

其他发现：LAB-Bench2 数据集本身也存在一些小错误

#位置问题性质影响
1cloning (gibson)克隆 CDS 还是全长转录本，题目未声明；且参考答案逐题不一致题目欠定义 + 标注不一致生物学最优答案被判错
2seqqa2 molecular_weightbenchmark 参考答案算错了：蛋白分子量多扣了一次水，参考值比真实值低 ~19%参考答案错误模型算对反被判错（蛋白题官方 0/8）
3seqqa2 primer_design 等目标基因藏在 580 kb 全基因组里，prompt 注入被截断注入方式对无工具模型不公平无工具模型理论上无法答对

问题 1 ·「CDS vs 全长」题目未声明克隆 CDS 还是全长

以第一题（cloning，id fb8fc27d…，Sorcs2）为例，原始 prompt 原话：

"replacing EGFP with the canonical mouse Sorcs2 transcript from Ensembl. I will amplify Sorcs2 from mouse cDNA …"

它只说「转录本 / 从 cDNA 扩增」，从未声明到底是把 CDS（ATG…stop） 还是整条转录本（含 5′/3′UTR）克隆进去。两者差异巨大：

插入物坐标(在 sorcs2-201-…gb)长度组装后全长
CDS（生物学标准做法）80..35593480 bp11671 bp
整条转录本（参考答案采用）1..57075707 bp13898 bp

问题 2 ·molecular_weight 蛋白题：benchmark 的参考答案算错了

benchmark 的标准答案本身算错。 molecular_weight 共 20 题，其中8 道蛋白题的参考分子量都给错了，系统性偏低约 19%。（另 6 道 DNA、6 道 RNA 不受影响。）

逐题核验（官方答案错误率）= 8/20 = 40%，且整齐地全部落在蛋白题：

类型题数官方答案错偏离标准值
protein88/8 全错14–18%（远超 1% 容差）
dna60/60.3–1.0%（容差内）
rna60/60.2–0.5%（容差内）

错误的参考值 vs 正确分子量（实测）：

蛋白序列benchmark 参考(错)正确分子量(ProtParam/Biopython)偏差
MKTLLLTLVV9501130.48−16% (−180 Da)
MFVFLVLLPLVSSQCVNLTT18632224.72−16% (−362 Da)
…（8 道蛋白题全部如此）偏低 ≈ 18×n Da——

用我们已落盘的答案重新判分：8 道蛋白题里，模型（只要给了答案）算出的都是正确分子量，却因为参考答案是错的而被官方判定全错（蛋白 0/8）。按正确分子量重判：

模型官方 molecular_weight 总分按正确答案纠正后其中蛋白题
flash 无工具11/2019/200/8 → 8/8
pro 代码 Agent12/2020/200/8 → 8/8
flash 代码 Agent6/2014/200/8 → 8/8

问题 3 ·primer_design：目标基因藏在 580 kb 基因组里，prompt 注入被截断

LAB-Bench2 seqqa2 的 inject 模式把题目相关文件全文贴进 prompt。但 primer_design（以及 amplicon_length / restriction_counts 等「指定基因名」的题）提供的文件是整条全基因组：

  • 文件 GCF_040556925.1_genomic.gbff = Mycoplasmoides genitalium 全基因组 580,076 bp；
  • 注入后 prompt = 200,033 字符（prompt_tok ≈ 73,406）；
  • 题目要的 rpsR 基因只是其中一小段，需要在这条巨长序列里定位。

文件因过长被从中间截断，模型根本看不到目标基因 rpsR，只能「猜」，随后给出的引物与标准答案不符，判 0。

这对不会调用工具的模型是结构性不公平：不是模型不会设计引物，而是它拿不到要设计引物的那段序列。
